# Celeste CLI v1.8 — Project Intelligence & MCP Server

**Date:** 2026-04-03
**Branch:** TBD (will be created at implementation time)
**Scope:** Project context (.grimoire), code graph indexing with LSH similarity, git-aware context, MCP server mode, web tools, cost tracking, hooks system, continuous learning/memory.

**Positioning:** Celeste is a full standalone agentic tool that also exposes itself as an MCP server. She works with any LLM provider (8+), escalates to stronger models for complex reasoning, and brings her own persona/voice to everything she does. Claude Code, Codex, or any MCP client can delegate tasks to her — especially content creation and development work that needs her personal branding.

## 1. `.grimoire` Project Context

### Problem

Celeste has no way to understand a project's conventions, structure, or rules without being told every session. Competitors use CLAUDE.md / .cursorrules for this. Celeste needs her own on-brand equivalent.

### Solution

`.grimoire` files — persona-themed project context files discovered automatically.

### Discovery (upward walk)

Celeste walks from cwd to filesystem root, collecting `.grimoire` files. Closer files have higher priority.

```
Priority (lowest -> highest):
1. ~/.celeste/grimoire.md              # User-global defaults
2. /repo-root/.grimoire               # Project-level
3. /repo-root/.celeste/grimoire/      # Directory of grimoire fragments (*.md)
4. /repo-root/.grimoire.local         # Private per-project (gitignored)
```

### File Format

Markdown with persona-flavored sections:

```markdown
# Grimoire: celeste-cli

## Bindings
- This is a Go project using Bubble Tea for TUI
- Module path: github.com/whykusanagi/celeste-cli
- Test command: go test ./cmd/celeste/... -count=1
- Lint command: golangci-lint run ./cmd/celeste/...

## Rituals
- Always run tests before committing
- Use conventional commit messages (feat/fix/refactor/docs)
- Never commit .env or config files with real keys

## Incantations
@./docs/ARCHITECTURE.md
@./docs/STYLE_GUIDE.md

## Wards
- Do not modify cmd/celeste/prompts/celeste.go without explicit permission
- Do not refactor the TUI state machine without a plan

## Hooks

### PreToolUse
- bash: cd {{workspace}} && go vet ./... 2>&1 | head -5

### PostToolUse
- write_file: cd {{workspace}} && gofmt -w {{path}}
- patch_file: cd {{workspace}} && gofmt -w {{path}}
```

**Sections:**
- **Bindings** -- project facts (language, structure, commands)
- **Rituals** -- behavioral rules (what to always/never do)
- **Incantations** -- `@path` includes, pulled into context (cycle detection, max depth 3)
- **Wards** -- protected areas the model should not touch unprompted
- **Hooks** -- pre/post tool execution commands (see Section 6)

### `@include` Support

`@./relative/path` or `@~/home/path` reads the file and inlines it. Circular references tracked and skipped. Binary files rejected. Total grimoire context capped at 25KB.

### CLI Commands

- `celeste init` -- generates a starter `.grimoire` with auto-detected project info
- `celeste grimoire` -- shows the resolved grimoire (all layers merged, includes expanded)
- `celeste grimoire --add` -- apply a suggested addition

### Package

```
cmd/celeste/grimoire/
├── discovery.go         # Walk upward, find .grimoire files
├── loader.go            # Parse sections, resolve @includes
├── merge.go             # Layer merging (user -> project -> local)
└── init.go              # celeste init auto-detection + template
```

## 2. Code Graph Index

### Problem

Without structural understanding of the codebase, the model wastes context reading files to orient itself. The common approach (grep) fails when the model doesn't know the right keywords — searching "authentication" misses `validateSession`, `checkExpiry`, `ErrUnauthorized`. Large files (e.g., a 3000-line main.go) blow up the context window when read directly. Cursor lacks a relationship view showing how code entities connect.

### Solution

A three-layer search strategy backed by a structural code graph in SQLite:

| Layer | Query Type | What It Finds | When |
|-------|-----------|---------------|------|
| **Graph** | "what calls X", "what implements Y" | Structural relationships | Model knows a specific symbol |
| **Semantic (MinHash + LSH)** | "code related to authentication" | Conceptually related symbols | Model has a concept, not a name |
| **Keyword** | "validateSession" | Exact matches | Model knows the exact name |

Each layer covers a different failure mode: graph fails when you don't know where to start, keyword fails when you don't know the vocabulary, LSH fills the gap with fuzzy conceptual search.

### Entity Model

| Entity | Examples | Stored Fields |
|--------|----------|---------------|
| **Symbol** | function, type, interface, method, const, var | name, kind, file, line, signature, package |
| **Edge** | calls, imports, implements, embeds, references | source -> target, kind |
| **File** | source files | path, language, size, content hash |

### Parsing Strategy

**Go projects** (deep parsing via `go/ast` + `go/types`):
- Full symbol extraction: functions, types, interfaces, methods, consts
- Call graph edges, interface implementation edges, import graph, embedded struct relationships

**Other languages** (regex-based extraction):
- Function/method declarations, import/require statements, export declarations
- No call graph (would require tree-sitter / CGo)

Language detected from file extensions + manifest files (`go.mod`, `package.json`, `Cargo.toml`, `pyproject.toml`).

### Semantic Code Search (MinHash + LSH, Pure Go)

The key insight: index symbols by **enriched shingles** derived from identifiers, types, body references, and comments — not just function names. This enables concept-based search without an embedding model, API calls, or external dependencies.

**Shingle generation per symbol:**

```go
func shinglesForSymbol(sym Symbol, source []byte) []string {
    shingles := []string{}

    // 1. Name parts (split camelCase/snake_case)
    //    "validateSession" -> ["validate", "session"]
    shingles = append(shingles, splitIdentifier(sym.Name)...)

    // 2. Parameter/return types
    //    "(token string, store SessionStore) (*User, error)"
    //    -> ["token", "string", "Session", "Store", "User", "error"]
    shingles = append(shingles, extractTypeTokens(sym.Signature)...)

    // 3. Referenced identifiers in body (top 20 by frequency)
    //    Body calls: checkExpiry, loadFromRedis, ErrUnauthorized
    //    -> ["check", "expiry", "load", "Redis", "Err", "Unauthorized"]
    shingles = append(shingles, extractBodyIdentifiers(source, sym)...)

    // 4. Package name
    //    -> ["auth"] or ["middleware"]
    shingles = append(shingles, splitIdentifier(sym.Package)...)

    // 5. Comment keywords (if present)
    //    // ValidateSession checks auth token validity
    //    -> ["checks", "auth", "token", "validity"]
    shingles = append(shingles, extractCommentTokens(source, sym)...)

    return deduplicate(lowercase(shingles))
}
```

**How search works:**

```
LLM asks: "find code related to authentication"
  -> Generate query shingles: ["authentication", "auth", "login", "session",
                                "token", "credential", "user", "verify"]
  -> MinHash the query
  -> Compare against all symbol signatures (Jaccard similarity)
  -> Return top-K matches ranked by similarity
  -> Model gets: validateSession, SessionStore, TokenValidator,
                  UserCredentials, middleware.RequireAuth
  -> NOT: authorizationHeader (HTTP util), AuthorName (git config)
```

**Implementation:** MinHash (N=128 hash signatures per symbol using `hash/maphash`). At query time, brute-force Jaccard comparison against all symbols — sub-10ms for projects up to 50k symbols. If scale demands it later, LSH bands (B bands of R rows) can be layered on top of existing MinHash signatures in ~50 lines to get O(1) approximate lookup.

**What makes this different from grep:** A search for "database connection pool" finds `initDBPool`, `pgxPoolConfig`, and `connectionManager` even though none match the exact query string. The enriched shingles capture semantic similarity through shared tokens, types, and referenced identifiers.

**What makes this different from embedding search:** No API calls to an embedding model, no vector database, runs entirely offline. Not as good at pure semantic leaps ("auth" = "login" with zero shared tokens) but dramatically better than grep and costs nothing to run.

### Storage

SQLite via `modernc.org/sqlite` (pure Go, no CGo) at `.celeste/codegraph.db`:

```sql
CREATE TABLE symbols (
    id INTEGER PRIMARY KEY,
    name TEXT NOT NULL,
    kind TEXT NOT NULL,
    package TEXT,
    file TEXT NOT NULL,
    line INTEGER,
    signature TEXT,
    minhash BLOB               -- N uint64 MinHash signatures
);

CREATE TABLE edges (
    source_id INTEGER REFERENCES symbols(id),
    target_id INTEGER REFERENCES symbols(id),
    kind TEXT NOT NULL
);

CREATE TABLE files (
    path TEXT PRIMARY KEY,
    language TEXT,
    size INTEGER,
    content_hash TEXT,
    indexed_at INTEGER
);

CREATE INDEX idx_symbols_name ON symbols(name);
CREATE INDEX idx_symbols_file ON symbols(file);
CREATE INDEX idx_edges_source ON edges(source_id);
CREATE INDEX idx_edges_target ON edges(target_id);
```

### Incremental Updates

On startup: walk file tree, compare content hashes, re-parse only changed/new files, remove symbols/edges for deleted files. First index: 5-15 seconds. Incremental: <1 second.

### Context Injection

The model does NOT get the full graph dumped into context. Celeste injects a project summary:

```
Project: celeste-cli (Go)
Packages: 15 | Files: 125 | Symbols: 1,847
Key packages: tools/ (412 symbols), tui/ (298 symbols), llm/ (187 symbols)
Entry points: cmd/celeste/main.go
```

The full graph is queried on-demand via three built-in tools:
- `code_search` -- semantic search ("find auth-related code") via MinHash similarity over enriched shingles. Also supports exact keyword search as fallback.
- `code_graph` -- structural queries: what calls this function, what implements this interface, dependency chains, symbol neighborhood (1-2 hops).
- `code_symbols` -- list symbols in a file/package without reading the full source. Solves the "main.go is 3000 lines" problem by showing the model what's in a file before deciding what to read.

### Package

```
cmd/celeste/codegraph/
├── index.go             # Indexer lifecycle (build, update, query)
├── parser_go.go         # Go AST parsing -> symbols + edges
├── parser_generic.go    # Regex-based extraction for other languages
├── shingle.go           # Enriched shingle generation from symbols
├── minhash.go           # MinHash signature computation (~100 lines)
├── similarity.go        # Jaccard similarity search over MinHash signatures
├── store.go             # SQLite schema + CRUD
└── detect.go            # Language detection from files/manifests
```

## 3. Git-Aware Context

### Startup Snapshot

On conversation start, 5 git commands run in parallel:

```
git rev-parse --abbrev-ref HEAD          -> current branch
git symbolic-ref refs/remotes/origin/HEAD -> main branch
git status --short                        -> staged/unstaged changes
git log --oneline -n 5                    -> recent commits
git config user.name                      -> committer name
```

Truncated to 2KB max. Injected into system prompt with the note: "This is a snapshot at conversation start and will not update during the conversation."

### Live Git Tools

Two new built-in tools for fresh git data:

- `git_status` -- `git status` + `git diff --stat`. `IsReadOnly=true`, `ConcurrencySafe=true`.
- `git_log` -- `git log` with configurable flags. `IsReadOnly=true`, `ConcurrencySafe=true`.

Always allowed (read-only, no permission prompt). Structured output (parsed, not raw shell).

### Diff Feedback

When `write_file` or `patch_file` executes, the tool result automatically includes a mini-diff showing what changed. Immediate feedback without running `git diff`.

## 7. Hooks System

### Configuration

Defined in `.grimoire` under `## Hooks`:

```markdown
### PreToolUse
- bash: cd {{workspace}} && go vet ./... 2>&1 | head -5

### PostToolUse
- write_file: cd {{workspace}} && gofmt -w {{path}}
- patch_file: cd {{workspace}} && gofmt -w {{path}}

### PreCommit
- *: cd {{workspace}} && go test ./... -count=1
```

### Hook Protocol

```go
type Hook struct {
    Event    string   // PreToolUse, PostToolUse, PreCommit
    Tool     string   // tool name to match, or "*" for all
    Command  string   // shell command with {{variable}} templates
    Timeout  int      // seconds, default 30
}

type HookResult struct {
    Decision string   // "approve" or "block"
    Output   string   // stdout/stderr
    ExitCode int      // 0 = approve, non-zero = block
}
```

- Exit 0 -> approve (continue)
- Exit non-zero -> block (cancel tool, show output to model)
- Timeout -> block with timeout error
- Template variables: `{{workspace}}`, `{{tool}}`, `{{path}}`, `{{command}}`

### Execution Flow

```
Tool call arrives
  -> Run PreToolUse hooks matching this tool
  -> If any hook blocks: return error to model, skip execution
  -> Execute tool
  -> Run PostToolUse hooks
  -> PostToolUse can trigger side effects (e.g., gofmt reformats)
  -> Return result to model
```

### Package

```
cmd/celeste/hooks/
├── hooks.go             # Hook execution engine
├── parser.go            # Parse hooks from .grimoire
└── template.go          # {{variable}} expansion
```

## 8. Continuous Learning (Memory System)

### Three Layers of Persistence

| Layer | Owner | Purpose | Scope |
|-------|-------|---------|-------|
| `.grimoire` | User | Project instructions | Per-project, version controlled |
| Memories | Celeste | Learned knowledge | Per-project at `~/.celeste/projects/<hash>/memories/` |
| Code graph | Machine | Structural understanding | Per-project at `.celeste/codegraph.db` |

### Memory File Format

```markdown
---
name: go-any-preference
description: Use any instead of interface{} -- Go 1.26 target
type: feedback
created: 2026-04-03
project: celeste-cli
---

Use `any` instead of `interface{}` throughout the codebase.

**Why:** Project targets Go 1.26 where `any` is preferred. Linter flags `interface{}`.

**How to apply:** All new code uses `any`. When editing existing files, replace opportunistically.
```

### Memory Types

| Type | What | Example |
|------|------|---------|
| `user` | Who you are, how you work | "Senior Go dev, prefers terse responses" |
| `feedback` | Corrections and confirmations | "Use any not interface{}" |
| `project` | Ongoing work, decisions | "Auth rewrite driven by compliance" |
| `reference` | Pointers to external systems | "Bugs tracked in Linear INGEST" |

### Memory Index (MEMORY.md)

Lightweight pointer file loaded into system prompt every conversation. Capped at 200 lines / 25KB.

```markdown
- [go-any-preference](go-any-preference.md) -- Use any instead of interface{}, Go 1.26
- [commit-style](commit-style.md) -- Conventional commits with co-author line
```

### When Memories Are Saved

**Explicit:** User says "remember this" -> save immediately.

**Heuristic extraction** (post-turn, no LLM call):
- Correction patterns ("no", "don't", "stop", "use X instead") -> `feedback` candidate
- Remember requests ("remember", "note that", "for next time") -> save immediately
- Decision patterns ("because", "we decided", "reason is") -> `project` candidate

Candidates shown to user for confirmation before saving. Celeste never silently memorizes.

### Memory Staleness

Memories older than 7 days get a verification step:
> "This memory is from 12 days ago. Verifying against current code before applying..."

If a memory references files/functions that no longer exist, Celeste flags it and offers to update or remove.

### Code Graph Auto-Update

1. **Startup** -- incremental (content hash comparison), <1 second
2. **After file mutations** (`write_file`, `patch_file`) -- re-index single file as PostToolUse hook, ~5ms
3. **After git state changes** (`git checkout/pull/merge/rebase` in bash) -- full incremental re-index

### Grimoire Suggestions

Celeste suggests `.grimoire` additions when she notices patterns:
- Same mistake corrected twice -> "Add a ward to your grimoire?"
- Convention discovered -> "Add to Bindings?"
- Successful verify command -> "Add as a Ritual?"

Never modifies `.grimoire` without asking.

### Memory CLI

```bash
celeste memories                          # list all for current project
celeste memories --global                 # list user-global memories
celeste memories --type feedback          # filter by type
celeste remember "always run go vet"      # save from command line
celeste forget go-any-preference          # delete a memory
```

### Package

```
cmd/celeste/memories/
├── store.go             # Read/write/delete memory files
├── index.go             # MEMORY.md management
├── extract.go           # Heuristic extraction from conversation turns
├── types.go             # Memory types, frontmatter parsing
```

## 10. Session Persistence & Resume

### Problem

Users can't pick up where they left off. Multi-day work requires starting fresh every session.

### Solution

Serialize full conversation state to disk. Resume any previous session by ID or interactive picker.

### Storage

Sessions persisted to `~/.celeste/sessions/<project-hash>/`:

```
~/.celeste/sessions/
├── a1b2c3d4/                          # project hash
│   ├── 2026-04-03T14-30-00.json       # session file
│   ├── 2026-04-02T09-15-00.json
│   └── latest -> 2026-04-03T14-30-00.json
```

Each session file contains:
```json
{
  "id": "2026-04-03T14-30-00",
  "project": "celeste-cli",
  "title": "v1.8 code graph implementation",
  "messages": [...],
  "metadata": {
    "provider": "grok",
    "model": "grok-4-1-fast",
    "totalCost": 0.47,
    "totalTokens": 24500,
    "filesModified": ["cmd/celeste/codegraph/index.go", "..."]
  }
}
```

### CLI & Commands

- `celeste resume` -- interactive picker showing recent sessions with title, date, message count
- `celeste resume <session-id>` -- resume specific session
- `celeste sessions` -- list all sessions for current project
- `/resume` slash command in TUI
- Auto-resume: on startup, offer to continue the most recent session if it's < 4 hours old

### Auto-Save

Session saved after every turn (not just on exit). Prevents data loss on crashes or force-quit.

## 11. File Checkpointing & Diff

### Problem

No safety net when the model modifies files. Can't review aggregate changes before committing. Silent overwrites if another tool/editor modifies a file mid-session.

### Solution

Three features working together: file snapshots, stale detection, and session diff view.

### File Checkpointing

Before every `write_file` or `patch_file`, snapshot the original file:

```
~/.celeste/checkpoints/<session-id>/
├── cmd/celeste/main.go.1          # first version before edit
├── cmd/celeste/main.go.2          # second version before edit
└── cmd/celeste/tools/registry.go.1
```

- `celeste revert <file>` -- restore most recent checkpoint for a file
- `/undo` slash command -- revert the last file modification
- Checkpoints cleaned up after 7 days

### Stale File Detection

Track mtime when `read_file` reads a file. Before `write_file` or `patch_file`, compare current mtime against last-read mtime:

```go
type FileTracker struct {
    readTimes map[string]time.Time  // path -> mtime at last read
}

func (ft *FileTracker) CheckStale(path string) error {
    // If file was modified since last read, return error
    // Model must re-read the file before editing
}
```

|----------|---------------|-----|

| xAI (Grok) | `reasoning_effort` | API parameter |
| OpenAI (o-series) | `reasoning_effort` | API parameter |
| Others | N/A | Silently skip |

Add to `LLMBackend` interface:
```go
type ThinkingConfig struct {

    BudgetTokens int
}

// Passed via existing SendMessageStreamEvents context or as new parameter
```

### Thinking Content Display (Implementation Spec)

When extended thinking is enabled, providers return thinking content as a separate block type alongside the response. This content must flow through the entire pipeline — from backend streaming through to TUI display.

#### Backend Streaming

Each backend must handle thinking blocks in all three streaming methods:

**Anthropic:** Returns `"thinking"` content blocks via `content_block_start/delta/stop`. Thinking deltas contain raw reasoning text. Must be captured alongside `"text"` and `"tool_use"` blocks.

**Gemini:** Returns thinking content via `Candidate.Content.Parts` with a `Thought` field (boolean). Check `part.Thought == true` to distinguish thinking from response text.

**OpenAI (o-series):** Reasoning tokens are internal — the model doesn't expose reasoning text. Only token counts are available via `completion_tokens_details.reasoning_tokens`. Display token count only, not content.

**xAI (Grok):** Reasoning effort is a request parameter, but Grok does not return separate thinking text. Same as OpenAI — token counts only.

#### New Stream Events

Add a new `StreamEventType`:

```go
EventThinkingDelta StreamEventType = 5  // Thinking/reasoning content delta

type StreamEvent struct {
    // ... existing fields ...
    ThinkingDelta string  // for EventThinkingDelta
}
```

Backends emit `EventThinkingDelta` as thinking content streams in (Anthropic only, currently). Other backends emit nothing for thinking — it's internal.

#### TUI Message Types

```go
type ThinkingContentMsg struct {
    Content string  // accumulated thinking text
    Tokens  int     // thinking token count (for providers that report it)
}
```

#### TUI Display Component

`tui/thinking.go` — collapsible thinking panel:

```
▸ Thinking... (2.4k tokens)                    [click to expand]

▾ Thinking (2.4k tokens)                       [click to collapse]
│ I need to consider the implications of this
│ change across the codebase. The registry.go
│ file has ExecuteWithProgress which is called
│ from... [dimmed italic text, corrupted theme]
```

- Default: collapsed, showing only token count
- Toggle: Enter or click to expand/collapse
- Style: dimmed text, italic, lighter border (visual hierarchy below main response)
- Persona touch: corruption glitch effects on thinking text (she's "processing")
- Max display: 50 lines when expanded (scroll for more)

#### Context Bar Integration

When thinking is active, show thinking tokens separately:

```
 ◆ tokens: 12.4k / 128k ▓▓▓░░░░░░░ 9% │ think: 2.4k │ $0.03 │ turn: 7
```

#### Pipeline

```
Backend streaming
  → EventThinkingDelta emitted for each thinking chunk
  → TUI accumulates thinking content in ThinkingContentMsg

  → On EventMessageDone: finalize thinking display, show total tokens

```

#### What NOT To Do

- Don't send thinking content back to the LLM in follow-up messages — it's for display only
- Don't include thinking in session persistence (it's ephemeral reasoning)

## 14. Prompt Caching

### Problem

Long sessions with Claude/Gemini cost 5-10x more than necessary because the system prompt (persona + grimoire + tools) is re-sent every turn without cache hints.

### Solution

Structure the system prompt with cache-control breakpoints so providers can cache the static prefix.

### Strategy

```
System prompt structure:
┌─────────────────────────────────┐
│ Celeste persona prompt          │ ← CACHED (never changes)
│ Tool definitions (JSON schemas) │ ← CACHED (changes only when MCP tools change)
│ .grimoire content               │ ← CACHED (changes only on grimoire reload)
├─────────────────────────────────┤
│ Git snapshot                    │ ← NOT CACHED (per-session)
│ Code graph summary              │ ← NOT CACHED (can change)
│ Memory (MEMORY.md)              │ ← NOT CACHED (can change)
│ Current date                    │ ← NOT CACHED (changes daily)
└─────────────────────────────────┘
```

### Implementation

For Anthropic: use `cache_control: {"type": "ephemeral"}` breakpoints in the system prompt blocks.

For Gemini: use `cached_content` API for the static prefix.

For other providers: no action needed (no cache API).

### Cost Impact

Typical session: 80% of system prompt is static. Cache hit rate >95% after first turn. Cost reduction: 50-90% on input tokens for Claude/Gemini providers.

## 18. Graceful Interrupt Handling

### Problem

Ctrl+C during a long operation (LLM streaming, tool execution) should abort the current work without killing the session.

### Solution

Two-level interrupt:

| Action | Behavior |
|--------|----------|
| **Ctrl+C once** | Cancel current LLM stream or tool execution. Partial results preserved. Model receives abort notification. Session continues. |
| **Ctrl+C twice** (within 2s) | Exit the session. Auto-save triggers. |
| **Escape** | Cancel current input or dismiss active modal/panel. |

### Implementation

```go
// In TUI Update loop
case tea.KeyMsg:
    if msg.Type == tea.KeyCtrlC {
        if m.interruptPending && time.Since(m.lastInterrupt) < 2*time.Second {
            // Double Ctrl+C: exit
            return m, tea.Quit
        }
        m.interruptPending = true
        m.lastInterrupt = time.Now()
        // Cancel current context
        if m.cancelFunc != nil {
            m.cancelFunc()
        }
        // Show "press Ctrl+C again to exit"
    }
```

Tools with `InterruptBehavior = InterruptBlock` (e.g., file writes mid-operation) finish before the abort takes effect. Tools with `InterruptCancel` abort immediately.

## 22. Implementation Patterns & Gotchas

Critical architectural patterns learned from reference implementation. These prevent subtle bugs.

### Session Persistence

- **Ephemeral messages**: Tool progress messages (e.g., bash streaming output, spinner updates) must NOT be persisted or sent on resume. They corrupt the message chain because the LLM expects tool_result to follow tool_use, not interleaved progress. Filter these out during serialization.
- **Auto-save on every turn**: Don't wait for exit. Crashes, force-quit, and OOM kills lose everything if sessions only save on clean exit.
- **JSONL format**: One message per line. Allows append-only writes (no full-file rewrite on each turn) and easy truncation for corruption recovery.

### File Checkpointing

- **Two-phase backup**: (1) Read mtime, (2) create backup, (3) re-check mtime before committing backup reference. This prevents a race where an external editor modifies the file between step 1 and step 2, causing the backup to capture the wrong version.
- **Null sentinel for new files**: When a file didn't exist before the edit, store `null` as the backup path (not an empty file). This distinguishes "file was empty" from "file didn't exist" during revert.
- **Max 100 snapshots**: Cap per session to prevent disk bloat on long agent runs.

### Plan Mode

- **Plan file read at exit time, not enter time**: Store the plan path, not the plan content. This lets users edit the plan in their IDE between `/plan` and `/plan execute`. Reading at enter time would freeze the plan.
- **Restrict tools, not the model**: Plan mode doesn't change the system prompt or model behavior — it restricts which tools the permission system allows. The model can still reason freely; it just can't execute writes.

### Prompt Caching

- **Static prefix, dynamic suffix**: Structure the system prompt so everything stable (persona, tool schemas, grimoire) is at the top, and everything volatile (git status, date, memories) is at the bottom. Cache breakpoints go between the two regions.
- **Per-tool schema hashing**: When checking if the cache broke, don't just hash all tool schemas together. Hash each tool individually. 77% of cache breaks in the reference are caused by a single MCP tool changing its schema — per-tool hashing tells you which one.
- **Compaction resets cache baseline**: After compacting history, the cache stats will show a "break" (fewer cached tokens). Don't alert on this — it's expected. Track compaction events and suppress false positives.

### Subagent Spawning

- **Identical placeholders for cache sharing**: When spawning multiple subagents, the tool_result placeholders for parent tool calls should be byte-identical across all children. This maximizes prompt cache hit rate because all children share the same message prefix up to their individual directive.
- **Block recursive spawning**: Detect when running inside a subagent (via a marker in messages) and reject spawn attempts. Unbounded recursion is an easy footgun.
- **Inherit rendered system prompt, not raw config**: Pass the parent's already-rendered system prompt bytes to the child, don't re-render from config. Re-rendering can produce different output (timestamps, git status changes) which busts the cache.

### Extended Thinking

- **Thinking and effort are separate knobs**: Thinking budget controls how many reasoning tokens the model uses. Effort controls overall computation quality. They're orthogonal — a model can have high effort but no thinking budget (or vice versa).
- **Session-scoped max**: The "max" effort level should not persist across sessions for non-power-users. It's expensive and usually requested for a specific task, not as a default.
- **Provider-specific parameters**: Each provider has a different API field for thinking/reasoning. Map them explicitly rather than hoping for compatibility.

### Graceful Interrupt

- **Don't claim Ctrl+C when idle**: In a terminal, Ctrl+C at an empty prompt should be handled by the terminal (for copy-selection, etc.), not by the app. Only intercept Ctrl+C when a task is actively running.
- **Clear permission queue on interrupt**: If the model requested 5 tool calls and the user interrupts after 2, clear the remaining 3 from the permission queue. Don't prompt for tools that won't run.
- **3-second window for double-press**: The window between first Ctrl+C (abort task) and second Ctrl+C (exit app) should be ~3 seconds. Too short = accidental exits. Too long = frustrating when you actually want to quit.

### General Patterns

- **Lazy evaluation with caching**: Plan slugs, diff computations, system prompt sections — compute on-demand and cache for the session. Avoid expensive operations during render cycles.
- **AbortSignal threading**: Pass `context.Context` (Go's abort signal equivalent) through every tool execution, hook call, and subagent spawn. Coordinated shutdown requires the signal to reach every running goroutine.
- **Last-write-wins for concurrent state**: Task mutations, memory writes, and checkpoint operations in a multi-agent environment use file locks with retry/backoff, not distributed transactions. Eventual consistency is acceptable for these use cases.

### MCP Agent Code Quality

When Celeste edits code via the MCP agent pipeline (JSON-RPC → agent → tool calls → write_file), two systematic issues arise:

- **Double JSON escaping**: Strings like `\n` in Go source get escaped to `\\n` because they pass through two JSON encode/decode cycles (MCP transport + tool argument serialization). The `write_file` and `patch_file` tools should detect and unescape double-escaped content, or the agent runtime should handle raw string content without re-encoding.
- **Import grouping**: Go convention requires stdlib imports grouped separately from third-party imports. Models consistently miss this. Solution: add `goimports -w` as a default PostToolUse hook for `.go` files in the `.grimoire` Hooks section. This is cheaper than teaching every model Go import conventions.
- **gofmt enforcement**: Always run `gofmt -w` via PostToolUse hook on Go files. This catches formatting issues before they reach CI.
