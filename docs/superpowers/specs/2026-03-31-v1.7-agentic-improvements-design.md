# Celeste CLI v1.7 — Agentic Improvements Design Spec

**Date:** 2026-03-31
**Branch:** `feat/v1.7-agentic-improvements`
**Scope:** Bottom-up rebuild of tool execution, context management, permissions, MCP support, and TUI enhancements — with full feature parity to v1.6.4.

**Reference material:** `~/Downloads/agentic_code` — a TypeScript/TSX agentic CLI framework demonstrating streaming tool execution, multi-agent orchestration, MCP integration, and sophisticated terminal UX.

## 1. Unified Tool Abstraction Layer

### Problem

Celeste v1.6.4 has two separate tool systems:
- `skills/` — 21 built-in skills for Chat/Claw mode (weather, tarot, crypto, etc.)
- `agent/dev_skills.go` — 6 dev tools for Agent mode (read_file, write_file, run_command, etc.)

These share no code, have different interfaces, and cannot be mixed across modes.

### Solution

A single `tools/` package with a unified `Tool` interface that works across all runtime modes.

### Tool Interface

```go
type Tool interface {
    Name() string
    Description() string
    Parameters() json.RawMessage           // JSON Schema for LLM function calling
    IsConcurrencySafe(input map[string]any) bool
    IsReadOnly() bool
    ValidateInput(input map[string]any) error
    Execute(ctx context.Context, input map[string]any, progress chan<- ProgressEvent) (ToolResult, error)
    InterruptBehavior() InterruptBehavior  // Cancel vs Block
}

type InterruptBehavior int
const (
    InterruptCancel InterruptBehavior = iota  // Ctrl+C aborts execution
    InterruptBlock                            // Ctrl+C blocked until completion
)

type ToolResult struct {
    Content  string
    Error    bool
    Metadata map[string]any
}

type ProgressEvent struct {
    ToolName string
    Message  string
    Percent  float64  // -1 for indeterminate
}
```

### Package Structure

```
tools/
├── tool.go              # Tool interface + ToolBuilder helper
├── registry.go          # Global registry, mode filtering, custom tool loading
├── executor.go          # StreamingToolExecutor
├── progress.go          # ProgressEvent types
├── builtin/             # All built-in tool implementations
│   ├── bash.go          # Shell execution (from dev_run_command)
│   ├── read_file.go     # File reading (from dev_read_file)
│   ├── write_file.go    # File writing (from dev_write_file)
│   ├── patch_file.go    # Surgical edits (from dev_patch_file)
│   ├── list_files.go    # Directory listing (from dev_list_files)
│   ├── search.go        # Text search (from dev_search_files)
│   ├── weather.go       # Weather skill (migrated from skills/)
│   ├── tarot.go         # Tarot skill (migrated)
│   ├── crypto.go        # Crypto skills (migrated)
│   ├── media.go         # Image gen/upscale (migrated from venice/)
│   ├── currency.go      # Currency conversion (migrated)
│   ├── encoding.go      # Base64 encoding (migrated)
│   ├── hash.go          # Hashing (migrated)
│   ├── uuid.go          # UUID generation (migrated)
│   ├── password.go      # Password generation (migrated)
│   ├── reminder.go      # Reminders (migrated)
│   ├── notes.go         # Notes (migrated)

```

### Mode Filtering

```go
func (r *Registry) GetTools(mode RuntimeMode) []Tool
```

- `ModeChat` — all tools (built-in + MCP + custom)
- `ModeClaw` — all tools (same as chat, with tool loop)
- `ModeAgent` — dev tools + skills + MCP (filtered by agent context)
- `ModeOrchestrator` — routed per task lane

### Custom Tool Loading

Custom tools from `~/.celeste/skills/` are loaded at startup via `registry.LoadCustomTools()`. They implement the same `Tool` interface through a JSON definition + handler adapter (preserving current `skills.json` format).

## 2. Streaming Tool Executor

### Problem

Celeste v1.6.4 waits for the full LLM response, collects all tool calls, executes them in a batch, then sends results back. This adds latency — tools could start executing while the model is still generating.

### Solution

A `StreamingToolExecutor` that begins tool execution as tool_use blocks arrive from the streaming API.

### Execution Model

```
API streams tool_use blocks
  ↓
executor.AddTool(block)     ← non-blocking, called per block as it arrives
  ↓
Concurrency check:
  ├─ IsConcurrencySafe=true  → execute immediately (parallel goroutine)
  └─ IsConcurrencySafe=false → enqueue, wait for serial slot
  ↓
During execution:
  ├─ Progress events → TUI updates in real-time via channel
  ├─ On error → abort sibling tools (cascading failure prevention)
  └─ On interrupt (Ctrl+C) → check InterruptBehavior per tool
  ↓
Results buffered in original call order (regardless of completion order)
  ↓
executor.Results() yields ordered results for next LLM turn
```

### Tool States

`Queued → Executing → Completed | Failed | Aborted`

### LLM Backend Changes

`SendMessageStream` must yield partial `StreamEvent` types as they arrive:

```go
type StreamEvent struct {
    Type         StreamEventType
    ContentDelta string           // for ContentDelta
    ToolUseID    string           // for ToolUse* events
    ToolName     string           // for ToolUseStart
    InputDelta   string           // for ToolUseInputDelta (partial JSON)
    Input        map[string]any   // for ToolUseDone (complete parsed input)
    Usage        *TokenUsage      // for MessageDone
}

type StreamEventType int
const (
    EventContentDelta StreamEventType = iota
    EventToolUseStart
    EventToolUseInputDelta
    EventToolUseDone
    EventMessageDone
)
```

Each backend (OpenAI, Google, xAI) adapts its native streaming format into these events.

### What This Replaces

- `main.go` lines 443-486 (batch-execute-and-follow-up loop)
- `agent/runtime.go` execution phase (sync batch → streaming executor)
- Claw mode's `claw_max_tool_iterations` config (executor handles the loop natively)

## 3. Context Window Management

### Problem

Celeste v1.6.4 has basic token tracking (`config/context.go`) with an 80% threshold warning but takes no automatic action. Long sessions hit context limits and fail.

### Solution

Automatic history compaction with token budgets and tool result capping.

### Package Structure

```
context/
├── budget.go            # TokenBudget — per-turn + session tracking
├── compaction.go        # History compaction strategies
├── summarizer.go        # LLM-based conversation summarization
└── limits.go            # Tool result size capping
```

### Token Budget

```go
type TokenBudget struct {
    ModelLimit      int    // e.g., 128k for gpt-4o
    SystemPrompt    int    // measured once at session start
    ToolDefinitions int    // measured once (re-measured when MCP tools change)
    History         int    // running total, updated per turn
    Available       int    // ModelLimit - SystemPrompt - ToolDefinitions - History
    TurnCount       int
    CompactCount    int    // how many times we've compacted
}
```

### Compaction Strategies

| Strategy | Trigger | Action |
|----------|---------|--------|
| **Reactive** | Token usage hits 80% of window | Summarize oldest turns, keep recent N turns intact |
| **Proactive** | Every N turns (configurable, default 20) | Check budget, compact if approaching limits |
| **Snip** | Single tool result exceeds cap | Truncate, store full on disk, give model file path + preview |

### Summarization Flow

1. Budget check fires (reactive or proactive)
2. Select oldest turns up to a target token reduction
3. Send those turns to the LLM: "Summarize this conversation segment, preserving key decisions, file paths mentioned, and task context"
4. Replace selected turns with a single `[compact_summary]` message
5. Update budget counters

### Tool Result Capping

- Configurable max per result (default 32KB)
- Over-cap results written to `~/.celeste/tool-results/{session-id}/{tool-call-id}.txt`
- Model receives: truncated preview + file path + total size

### What This Replaces

- `config/context.go` — subsumed by `context/budget.go`
- `config/tokens.go` — moved into `context/budget.go`
- TUI 80% warning — replaced by automatic compaction

## 4. Permission & Safety System

### Problem

Celeste v1.6.4 has hardcoded workspace guards in agent mode (path validation, sudo blocking) but no granular per-tool permission model. Users cannot configure what's allowed/denied.

### Solution

Multi-layer permission system with configurable rules and denial tracking.

### Package Structure

```
permissions/
├── permission.go        # Core types: Decision, Rule, Mode
├── checker.go           # Multi-layer permission evaluation
├── rules.go             # Rule matching (tool name, input patterns)
├── denial.go            # Denial tracking + escalation
└── config.go            # Persistent rules (~/.celeste/permissions.json)
```

### Permission Modes

| Mode | Behavior |
|------|----------|
| `default` | Ask on destructive/write operations, auto-allow reads |
| `strict` | Ask on everything except explicitly allowed tools |
| `trust` | Auto-allow everything (current v1.6.4 behavior) |

### Evaluation Chain

```
Tool + Input
  ↓
1. Check alwaysDeny rules     → Deny (hard block, no prompt)
2. Check alwaysAllow rules    → Allow (skip further checks)
3. Check tool.IsReadOnly()    → Allow in default mode
4. Check pattern rules        → e.g., "bash(git *)" allowed, "bash(rm *)" denied
5. Fall through to mode       → default: prompt / strict: prompt / trust: allow
```

### Configuration (`~/.celeste/permissions.json`)

```json
{
  "mode": "default",
  "alwaysAllow": [
    "read_file",
    "list_files",
    "search",
    "bash(git *)",
    "bash(go test *)"
  ],
  "alwaysDeny": [
    "bash(sudo *)",
    "bash(su *)",
    "bash(rm -rf *)"
  ]
}
```

### Denial Tracking

- Counts consecutive denials per session
- After 3 denials of the same tool: suggest adding an `alwaysDeny` rule
- After 5 total denials in a session: suggest switching to `strict` mode

### Migration from Current Code

- `agent/dev_skills.go` workspace path validation → permission rules
- Hardcoded `sudo`/`su` blocking → default `alwaysDeny` rules
- Agent mode can run in `trust` for CI or `strict` for untrusted goals

## 5. MCP (Model Context Protocol) Client

### Problem

Adding new tool capabilities to celeste-cli requires compiling them into the binary. The reference codebase supports dynamic tool discovery from external MCP servers.

### Solution

MCP client that discovers and adapts external tools into the unified `Tool` interface.

### Package Structure

```
tools/mcp/
├── client.go            # MCP client — lifecycle, capability negotiation
├── transport.go         # Transport layer (stdio process spawn, SSE HTTP)
├── discovery.go         # Tool discovery from server
├── adapter.go           # MCP tool → tools.Tool interface adapter
└── config.go            # Server config from ~/.celeste/mcp.json
```

### Server Configuration (`~/.celeste/mcp.json`)

```json
{
  "servers": {
    "sqlite": {
      "command": "npx",
      "args": ["-y", "@anthropic/mcp-server-sqlite", "--db", "./data.db"],
      "transport": "stdio"
    },
    "github": {
      "url": "https://mcp.github.com/sse",
      "transport": "sse",
      "env": {
        "GITHUB_TOKEN": "${GITHUB_TOKEN}"
      }
    }
  }
}
```

### Lifecycle

1. App startup → load `mcp.json`
2. For each server: spawn process (stdio) or connect (SSE)
3. Initialize handshake (protocol version, capabilities)
4. Discover tools → convert to `tools.Tool` interface via adapter
5. Register in tool registry as `source="mcp:{server_name}"`
6. During session: MCP tools appear alongside built-in tools in LLM requests
7. Permission system applies equally (MCP tools default to "ask" mode)
8. Server health monitored (reconnect on failure)
9. App shutdown → graceful disconnect

### Design Decisions

- MCP tools go through the same permission system — no implicit trust
- MCP tools are `IsConcurrencySafe = false` by default (conservative)
- Tool definitions re-discovered on reconnect (servers can change tools dynamically)
- Environment variable expansion (`${VAR}`) in config for secrets

### Out of Scope for v1.7

- MCP Resources (read-only data endpoints)
- MCP Sampling (server-initiated LLM calls)
- MCP server authoring (celeste-cli as a server)
- OAuth flows for authenticated servers

## 6. TUI Enhancements

### Approach

Extend the existing Bubble Tea component hierarchy — no rewrite. New components surface the new subsystems.

### New Components

#### Tool Progress Indicators (`tui/tool_progress.go`)

Real-time execution status in the chat viewport:

```
┌─ search "authentication" ─────── ⣾ executing
│  Found 14 matches across 6 files...
└────────────────────────────────── 1.2s

┌─ read_file cmd/celeste/main.go ─ ✓ done
└────────────────────────────────── 0.3s

┌─ bash "go test ./..." ────────── ⣾ executing
│  === RUN   TestTokenBudget
│  === RUN   TestCompaction
└──────────────────────────────────
```

- Progress events stream from executor into chat viewport
- Concurrent tools shown stacked (all visible simultaneously)
- Collapsed to single-line summary after completion
- Failed tools show error inline with red styling

#### Permission Prompts (`tui/permissions.go`)

Inline dialog in chat flow (not modal overlay):

```
╭─ Permission Required ──────────────────╮
│  bash wants to run: rm -rf ./dist      │
│  Risk: destructive                     │
│                                        │
│  [a] Allow once                        │
│  [A] Always allow "bash(rm *)"         │
│  [d] Deny                             │
│  [D] Always deny "bash(rm *)"         │
╰────────────────────────────────────────╯
```

- Blocks only the requesting tool — other concurrent tools continue
- Single keypress response
- "Always" choices persist to `permissions.json`

#### Context Budget Bar (`tui/context_bar.go`)

Status bar between chat and input:

```
 ◆ tokens: 12.4k / 128k ▓▓▓░░░░░░░ 9%  │  compact: 0  │  turn: 7
```

- Color: green (<50%), yellow (50-80%), red (>80%)
- Shows compaction count
- Collapses on narrow terminals

#### MCP Panel (`tui/mcp_panel.go`)

Accessed via `/mcp` command:

```
╭─ MCP Servers ──────────────────────────╮
│  ● sqlite         3 tools    stdio     │
│  ● github         8 tools    sse       │
│  ○ slack          disconnected         │
│                                        │
│  Total: 11 external tools available    │
╰────────────────────────────────────────╯
```

- Server health indicators (● connected, ○ disconnected, ◌ connecting)
- Drill into server to see tools
- Header shows connected server count

### Modified Components

- `app.go` — new message types for permission dialogs, tool progress, context updates
- `chat.go` — tool progress streaming inline in viewport
- `streaming.go` — handle `StreamEvent` types from executor
- `styles.go` — new styles for permission/progress/context elements

### Unchanged Components

- `input.go`, `skills_panel.go`, `collections.go`, `menu.go`, `selector.go`, split panel
